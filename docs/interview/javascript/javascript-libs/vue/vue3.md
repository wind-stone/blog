# Vue 3

## Vue3 使用 proxy 来实现响应式，带来了哪些新的问题？

Vue 3 采用 `Proxy` 替代了 Vue 2 的 `Object.defineProperty`，虽然完美解决了对象新增属性、数组索引修改等经典问题，但也因为其底层机制的特性，带来了一些新的“响应式陷阱”和性能挑战。主要的新问题包括以下几个方面：

### 1. 响应式解构陷阱 (Destructuring)

这是 Vue 3 中最常见的错误之一。当你对 `reactive` 定义的对象进行解构时，解构出的变量会失去响应性，变成普通的值。

- **错误表现**：`const { count } = state;` 会导致 `count` 变成一个普通的数字，后续修改不会触发视图更新。
- **正确做法**：需要使用 `toRefs` 来保持响应式连接，例如 `const { count } = toRefs(state);`。

### 2. 整个对象直接赋值导致响应式丢失

如果你给一个 `reactive` 定义的对象重新赋值一个全新的对象，原本的响应式连接会断开。

- **错误表现**：`state = { list: [1, 2, 3] };` 会覆盖掉原本的 Proxy 引用，导致响应式丢失。
- **正确做法**：应该修改对象内部的属性，或者改用 `ref` 来定义整个对象（通过 `state.value = ...` 赋值）。

### 3. 原始引用 (Original Reference) 操作失效

当你把一个原始对象赋值给响应式变量后，如果继续操作那个“原始对象”，是不会触发视图更新的。

- **错误表现**：`const raw = { a: 1 }; const state = reactive(raw); raw.a = 2;` 这种操作绕过了 Proxy 拦截，视图不会更新。
- **正确做法**：不要保留对原始对象的引用，始终通过响应式代理对象（Proxy）来操作数据。

### 4. 原型链属性天然不响应

Vue 3 的 `Proxy` 只能拦截对目标对象**自身属性**的访问，无法捕获原型链上属性的读写操作。

- **错误表现**：使用 `Object.create` 或类继承后直接进行 `reactive` 包装，修改原型上的属性不会触发界面更新。
- **正确做法**：Vue 3 官方推荐通过组合式 API（Composables）和 `provide/inject` 来替代继承；如果必须用类，应将响应式数据显式声明在实例自身（如 `this.data = reactive({...})`）。

### 5. 对象身份比较失败

被 `Proxy` 包装的对象与原始对象是不相等的（`===` 比较会返回 `false`）。

- **错误表现**：在进行依赖严格比对的操作（如数组的 `.filter()` 或 `.map()`）时，可能会因为身份不一致导致逻辑失败。
- **正确做法**：最佳实践是不要保留对原始对象的引用，只使用响应式版本进行后续操作。

### 6. 超大型应用中的性能瓶颈

在数据量极大的场景下，`Proxy` 的递归代理机制会带来显著的性能开销。

- **具体表现**：初始化深层嵌套对象或大型数组时，CPU 占用飙升、首屏渲染延迟；频繁更新大型表格时，DOM 重排/重绘阻塞主线程，导致页面卡顿。
- **优化策略**：对非动态数据使用 `markRaw` 跳过响应式转换；对大型对象优先使用 `shallowRef` 或 `shallowReactive` 避免深层代理；通过 `toRaw` 获取原始数据进行批量修改后，再手动触发一次性更新。

## Vue 3 里 reactive 和 ref 有什么区别？

在 Vue 3 的 Composition API 中，`ref` 和 `reactive` 是定义响应式数据的两大核心基石。它们的核心区别在于**支持的数据类型**、**访问方式**以及**底层实现机制**。

以下是两者的详细对比与使用指南：

### 1. 核心差异对比

| 对比维度         | ref                                                                 | reactive                                                                    |
| :--------------- | :------------------------------------------------------------------ | :-------------------------------------------------------------------------- |
| **支持数据类型** | 全能型：支持基本类型（String, Number等）和引用类型（Object, Array） | 对象专家：仅限引用类型（Object, Array, Map, Set），传入基本类型会丢失响应式 |
| **JS 访问方式**  | 必须通过 `.value` 属性进行读写                                      | 像普通对象一样直接访问属性，无需 `.value`                                   |
| **模板访问方式** | 自动解包，直接使用变量名即可，无需 `.value`                         | 直接使用属性名                                                              |
| **底层实现**     | 包装基本类型，若传入对象，内部会自动调用 `reactive` 进行处理        | 基于 ES6 Proxy 实现，直接代理整个对象及其嵌套属性                           |
| **整体替换对象** | 支持：通过 `ref.value = 新对象` 替换，响应式不会丢失                | 不支持：直接重新赋值会覆盖 Proxy 引用，导致响应式丢失                       |
| **解构响应式**   | 解构后仍保留响应式（解构出的是 Ref 实例）                           | 直接解构会丢失响应式，必须配合 `toRefs` 使用                                |

### 2. 使用场景建议

在实际开发中，选择哪个 API 主要取决于你要管理的数据结构：

- **推荐使用 `ref` 的场景**：
    - **基本类型数据**：如计数器、开关状态（Boolean）、输入框的值等。
    - **需要整体重置/替换的数据**：例如从后端接口获取列表后，需要直接 `list.value = res.data` 覆盖原数据。
    - **获取 DOM 元素**：替代 Vue 2 的 `this.$refs`，通过 `<div ref="domRef"></div>` 获取真实 DOM 节点。
- **推荐使用 `reactive` 的场景**：
    - **复杂业务模型**：包含多个相互关联属性的大对象，如用户信息、表单整组数据、配置项等。
    - **追求原生代码感**：不希望在逻辑代码中到处写 `.value`，希望像操作普通 JS 对象一样直接修改属性。
    - **聚合数据**：将一类变量聚合在一个对象中统一管理，减少零散变量的声明。

### 3. 高频易错点提醒

- **`reactive` 整体赋值陷阱**：使用 `reactive` 时，切忌直接对变量重新赋值（如 `state = { count: 1 }`），这会断开响应式连接。如需整体替换，应改用 `ref`，或使用 `Object.assign(state, { count: 1 })` 进行原地修改。
- **解构丢失响应式**：对 `reactive` 对象进行解构（如 `const { count } = state`）会导致 `count` 变成普通变量。如需解构并保持响应式，必须使用 `toRefs`（如 `const { count } = toRefs(state)`）。
- **忘记 `.value`**：在 `<script>` 逻辑代码中操作 `ref` 数据时，极易忘记写 `.value`，导致数据没有真正被修改（注意：在 `<template>` 模板中 Vue 会自动解包，无需写 `.value`）。
