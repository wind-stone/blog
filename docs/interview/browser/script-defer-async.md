# script 脚本 defer 和 async

HTML 里，`script` 标签添加 `defer` 和 `async` 的区别？

简单来说，`defer` 和 `async` 都是用来优化页面加载的，它们让脚本的**下载**过程不阻塞 HTML 解析，但核心区别在于**执行时机**和**执行顺序**。

## 核心区别一览

| 特性              | `defer`                                                        | `async`                                                    |
| :---------------- | :------------------------------------------------------------- | :--------------------------------------------------------- |
| **执行时机**      | HTML 解析**完全结束**后，`DOMContentLoaded` 事件**之前**执行。 | 脚本**下载完成后立即执行**，此时 HTML 可能还没解析完。     |
| **执行顺序**      | **保证顺序**，严格按照在 HTML 中出现的顺序执行。               | **不保证顺序**，谁先下载完谁先执行。                       |
| **与 DOM 的关系** | 执行时 DOM 已就绪，可以安全操作。                              | 执行时 DOM 可能未就绪，操作 DOM 有风险。                   |
| **适用场景**      | 需要操作 DOM、或有依赖关系的脚本（如 jQuery 和依赖它的插件）。 | 完全独立的脚本，不依赖其他脚本或 DOM（如统计、广告代码）。 |

### 执行流程对比

可以想象成浏览器在解析 HTML 时，遇到了带这些属性的 `<script>` 标签：

- **普通脚本（无属性）**：遇到就停下，下载完并执行完，再继续解析。**阻塞**。
- **`defer`**：后台下载，不阻塞解析。等整个 HTML 文档解析完成后，再按顺序执行这些脚本。
- **`async`**：后台下载，不阻塞解析。但一旦下载完，立刻暂停 HTML 解析，优先执行这个脚本。

### 如何选择？

- **优先用 `defer`**：这是现代 Web 开发最推荐的默认选项。如果你的脚本需要操作页面元素，或者多个脚本之间有依赖关系（比如先加载库，再加载业务代码），`defer` 是安全且可预测的选择。
- **特定场景用 `async`**：只适合那些完全独立、不依赖任何东西的“工具人”脚本。比如第三方流量统计、广告展示等，它们早一点晚一点执行都不影响页面核心功能。
- **不要同时使用**：如果同时写了 `async` 和 `defer`，现代浏览器会忽略 `defer`，只采用 `async` 的行为。
