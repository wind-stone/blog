# compact

在 Claude Code 中，`/compact` 是一个非常实用且关键的内置命令，主要用于**智能压缩当前的对话历史**。它不仅能有效管理上下文，还能显著降低使用成本并提升 AI 的响应质量。

以下是对 `/compact` 命令的详细说明：

## 1. 核心机制：它是如何工作的？

当你执行 `/compact` 时，Claude Code 并不会简单粗暴地清空聊天记录（那是 `/clear` 命令的工作）。相反，它会在后台执行一个“智能总结”操作：

- **提取精华**：回顾之前的对话、代码片段和当前任务状态，提取出核心上下文、已解决的问题、待办事项和重要的环境信息。
- **生成摘要**：将这些精华浓缩成一段精简的 Prompt，注入到新的对话轮次中。
- **丢弃冗余**：自动丢弃那些冗长的、逐字逐句的历史对话记录，以及早期探索过程中的中间结果和已解决的错误排查细节。

## 2. 语法与自定义指令

`/compact` 支持附加可选的“指令（instructions）”参数，允许你指定压缩时的侧重点，确保重要内容不被丢弃。

- **基本用法**：直接输入 `/compact`，Claude 会自行总结关键信息。
- **带提示压缩**：例如输入 `/compact "保留认证模块的讨论"` 或 `/compact 保留尚未解决的问题`，Claude 在总结时会重点保留你指定的内容。

## 3. 为什么要使用 `/compact`？（三大核心优势）

- **显著降低 Token 成本**：历史对话越长，每次提问携带的输入 Token 就越多。`/compact` 能瞬间将庞大的上下文“瘦身”，大幅减少后续请求的输入 Token 量（实测可减少 30%–50% 的输入消耗）。
- **提升响应速度**：大模型处理大量输入文本需要耗费计算资源。压缩历史记录后，首字响应时间会明显加快。
- **减少 AI 的“幻觉”和干扰**：如果之前的对话中经历了多次试错、报错和方向调整，这些“废弃”的思路会残留在上下文中，干扰模型后续的判断。`/compact` 能够“洗掉”这些噪音，让 Claude 保持专注。

## 4. 最佳使用时机

掌握正确的时机能让这个命令发挥最大效用：

- **完成独立模块后**：刚搞定一个复杂的 Bug 或写完一个完整函数，准备进入下一个不相关的子任务时。
- **陷入死胡同时**：如果发现 Claude 一直在原地打转、重复之前的错误代码，说明上下文已被污染。此时执行 `/compact` 可强制它跳出思维定势。
- **收到上下文警告时**：当上下文占用超过 80%，或收到「context is getting large」警告时，主动压缩是保持顺畅体验的关键。

## 5. ⚠️ 隐藏成本与避坑指南

虽然 `/compact` 非常有用，但在实际使用中需要注意其隐藏的 Token 成本：

- **全价计费陷阱**：`/compact` 触发的总结请求使用的是专门的总结系统提示（不带工具定义），这意味着**整个对话历史在执行 `/compact` 的这一次会按全价计费**。
- **最佳实践**：不要积累几十轮对话后再去 compact。建议在每完成一个子任务后就执行一次，这样被全价计费的内容会更少。
- **自动压缩的风险**：在长时间运行的会话中，如果触发了自动压缩，模型可能会因为无法准确预测你下一步的工作方向而偏离重点（例如把另一个警告当成无关信息丢弃）。因此，**带着明确的目标描述主动进行手动压缩**，往往比依赖自动压缩更安全。

## 6. `/compact` 与 `/clear` 的区别

- **`/compact`**：保留上下文的精华（摘要），适合当前任务还没完成、不想开新会话，或者需要延续长对话的场景。
- **`/clear`**：完全清空当前会话的对话历史和上下文，相当于开启一个全新会话。适合切换到完全不相关的新任务，或上下文出现严重混乱需要“重启”时。
