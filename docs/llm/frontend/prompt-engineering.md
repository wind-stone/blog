# 提示工程（Prompt Engineering）

## 谈谈你对 prompt engineering 在前端层面的理解，前端可以做哪些工作来提升模型输出质量？

在前端开发中，Prompt Engineering（提示工程）的本质是**前端逻辑与 AI 能力的结合点**。它不再仅仅是算法工程师的专属，而是前端开发者将模糊的用户需求转化为 AI 可理解指令，并控制模型输出以适配前端交互的核心手段。

前端开发者可以通过以下几个维度的工作，显著提升 AI 模型的输出质量和应用的稳定性：（每个维度下都有对应的示例，这些示例是围绕前端常见的“AI 辅助代码生成”与“数据交互”场景展开）

### 🎯 1. 设计结构化的 Prompt 模板

模糊的意图会导致 AI 输出不可控。前端需要将用户的自然语言需求，转化为包含明确要素的结构化 Prompt。一个高质量的前端 Prompt 通常包含以下要素：

- **角色设定**：告诉 AI 它的身份（如“你是一名资深前端工程师”）。
- **任务描述**：明确需要完成的具体任务。
- **上下文信息**：提供相关的技术栈、代码片段或背景资料。
- **约束条件**：明确必须遵守的规则（如“兼容 ES5”、“使用 Tailwind CSS”）。
- **输出要求**：指定输出的格式、长度和风格。

**场景**：前端需要调用 AI 生成一个带有特定技术栈要求的 React 组件。
**案例**：与其简单地说“帮我写一个商品列表组件”，不如构建一个包含角色、上下文、约束和输出要求的完整 Prompt 模板：

```text
【角色设定】
你是一名拥有10年经验的高级前端工程师，精通 React 生态和性能优化。

【任务描述】
请为我开发一个电商平台的“商品列表”组件。

【上下文与技术栈】
- 框架：Next.js 14 (App Router) + TypeScript
- 样式：Tailwind CSS
- 状态管理：Zustand
- 数据获取：TanStack Query

【约束条件】
1. 必须使用函数式组件和 Hooks，禁止使用 Class 组件。
2. 必须包含完整的 TypeScript 类型定义，禁止使用隐式 any。
3. 实现响应式网格布局（移动端单列，桌面端四列）。
4. 图片必须实现懒加载，并使用骨架屏（Skeleton）处理加载状态。

【输出要求】
请输出完整的组件代码，并为关键逻辑添加清晰的中文注释。
```

### 🎨 2. 严格约束输出格式（降低前端解析成本）

为了让 AI 的输出能无缝集成到前端页面中，必须对输出格式进行强约束，避免大量的后处理工作。

- **指定数据格式**：明确要求 AI 返回 `JSON`、`Markdown` 或特定的 XML 标签格式。例如：“请以 JSON 格式输出，包含 `name` 和 `age` 字段”。
- **控制随机性（Temperature）**：在调用模型 API 时，前端可以通过设置较低的温度参数（如 0.1-0.3），让模型的输出更加确定和一致，这对于代码生成和格式化输出至关重要。

**场景**：前端调用 AI 进行代码审查，需要将审查结果直接解析并渲染在页面上。
**案例**：强制要求 AI 返回标准的 JSON 格式，并配合“负向约束”杜绝多余的解释性文字，这样前端可以直接 `JSON.parse()` 使用：

````text
你是一名严格的前端代码审查员。请分析以下 React 代码，重点关注性能问题（如不必要的重渲染、useMemo 滥用等）。

【待审查代码】
{{paste_your_code_here}}

【输出格式要求】
请严格按照以下 JSON 数组格式输出，不要包含任何 Markdown 标记（如 ```json），也不要输出任何额外的解释性文字：
[
  {
    "line": 问题所在的行号,
    "severity": "严重/警告/建议",
    "issue": "问题的简短描述",
    "fix": "修复后的代码片段或建议"
  }
]

【负向约束】
- 禁止输出 JSON 以外的任何内容。
- 如果没有发现问题，请返回空数组 []。
````

### 📚 3. 运用 Few-shot Learning（少样本学习）

如果希望 AI 的输出风格或逻辑高度对齐业务需求，可以在 Prompt 中提供少量高质量的输入输出示例。AI 具备强大的上下文学习能力，通过几个示例就能快速模仿出期望的模式，而无需对模型进行额外的微调。

**场景**：希望 AI 将自然语言的设计需求，转化为团队内部约定的特定布局约束代码（DSL）。
**案例**：通过提供 2-3 个“输入 -> 输出”的标准示例，让 AI 快速学会团队内部的代码规范：

```text
请将以下自然语言描述的前端布局需求，转换为对应的 CSS 约束代码。请完全模仿以下示例的格式：

【示例 1】
输入：搜索框紧贴顶部，并且距离左右两边各 16 像素
输出：
searchBox.top = container.top + 0;
searchBox.left = container.left + 16;
searchBox.right = container.right - 16;

【示例 2】
输入：提交按钮在容器底部居中，距离底部 20 像素
输出：
submitButton.bottom = container.bottom - 20;
submitButton.centerX = container.centerX;

【现在请处理】
输入：取消按钮在提交按钮的左侧，两者间隔 12 像素，且垂直居中对齐
输出：
```

### 🛡️ 4. 模块化拆分与边界控制（保障稳定性）

在 AI 辅助编程或生成复杂页面时，一次性生成大量内容容易导致“上下文漂移”，甚至出现 AI 误改无关文件的情况。

- **单文件/单组件生成**：严格限制 Prompt 的作用域。例如，明确指令“只生成 `Header.tsx` 文件，不要修改其他文件”，能极大提升生成的稳定性和安全性。
- **负向提示（Negative Prompting）**：明确告诉 AI **不要**做什么。例如，“不要使用过时的 API”、“不要包含任何解释性文字，只输出代码”。

**场景**：使用 AI 编程助手（如 Cursor、Copilot）重构或生成多文件项目时，防止 AI “自作主张”修改无关文件。
**案例**：在生成复杂页面时，将任务拆解，并在 Prompt 中明确划定修改边界：

- **第一步（生成 Header）**：
  “请生成一个 Steam 风格的深色导航栏组件。**只生成 `Header.tsx` 文件，不要修改项目中的其他任何文件（如 `package.json` 或全局样式）**。”
- **第二步（生成 Hero Banner）**：
  “请生成一个带自动轮播功能的大背景图组件。**只生成 `HeroBanner.tsx` 文件，不要修改其他文件**。”

通过这种“单文件生成模式”，可以有效避免 AI 在生成代码时发生上下文漂移，甚至误删、覆盖已有配置的风险。

### 🔄 5. 结合后处理与人工校验（Answer Engineering）

Prompt Engineering 不仅在于输入，也在于对输出的优化。

- **输出解析与清洗**：前端可以通过正则表达式或解析器，从 AI 返回的文本中提取出纯净的代码或数据，过滤掉多余的解释性语句。
- **迭代优化**：Prompt 的编写是一个动态迭代的过程。前端需要根据实际渲染效果和用户反馈，不断调整 Prompt 的措辞和约束条件，甚至引入人工审核机制来补充特殊场景的训练数据。

**场景**：AI 生成的代码虽然大致正确，但可能包含一些不符合团队 ESLint 规范的细节，或者需要自动注入到现有页面中。
**案例**：前端不仅仅是接收 AI 的文本，还可以编写脚本对输出进行二次加工：

1. **代码自动清洗**：
   AI 有时会忽略分号或特定的引号风格。前端可以接入 `Prettier` 或 `ESLint --fix` 的 API，在 AI 返回代码字符串后、展示给用户之前，自动运行一遍格式化工具，确保代码风格与项目完全统一。
2. **正则提取与注入**：
   如果 AI 返回的内容是“代码 + 解释”的混合体，前端可以通过正则表达式（如 `/```tsx([\s\S]*?)```/`）精准提取出代码块部分，然后自动将其插入到编辑器（如 Monaco Editor）的指定光标位置，极大提升开发者的编码效率。

### 总结

**总结来说**，前端在提升模型输出质量上的核心工作，就是通过**结构化的输入设计**、**严格的格式与边界约束**以及**灵活的后处理机制**，将概率性的 AI 输出转化为稳定、可控、可直接落地的前端功能。
